## The Paradox

In 2025, a team of researchers from METR and RAND set out to measure something the industry had been assuming for years: that AI tools make developers faster. They ran a careful controlled study with experienced engineers on real-world tasks. The result was not what anyone expected.

Developers using AI coding tools were 19% slower than those working without them.

Let that sit for a moment. Not slower because the tools were buggy or the participants unfamiliar with them. The study controlled for that. These were experienced practitioners using tools they knew, on tasks that seemed well-suited to AI assistance. And they came out behind.

The finding didn't generate the backlash you might expect. The industry mostly absorbed it as a curiosity, a footnote, perhaps an edge case. But I think it's the most instructive data point about AI and software development that's been published in years. Not because it proves AI tools don't work — it doesn't — but because it reveals something important about how the industry has been thinking about them.

---

## What the Study Actually Measured

The METR study wasn't measuring whether AI could write code. It clearly can. What it measured was developer output in context: real tasks, real timelines, real tradeoffs. And in that context — which is the only context that matters — AI assistance introduced enough friction to produce a net slowdown.

The friction is instructive. Experienced developers don't just write code. They evaluate what they're reading, make architectural decisions, anticipate failure modes, and hold a mental model of a system that took months to build. When AI generates a function, a senior engineer doesn't just accept it. They read it, consider it, test it against their mental model, and often rewrite it. That process takes time.

Separately, research shows that code review time has increased 91% even as AI generates more code. The volume goes up; the scrutiny required to match it goes up proportionally. Meanwhile, 45% of AI-generated code fails security tests in controlled assessments, and post-hoc analysis finds roughly 1.7x more issues in AI-generated code compared to human-written code at similar feature complexity.

None of this means the tools are bad. It means the feedback loop is longer than advertised, and the costs land somewhere different than where the marketing points.

---

## What the Industry Got Wrong

The bet the industry made was essentially this: developers are slow because typing is slow. Therefore, automate typing. Ship copilots that complete lines, suggest functions, generate boilerplate. Make the fingers move faster.

This was a reasonable hypothesis in 2022. It turned out to be wrong.

Typing was never the bottleneck. Research on how developers actually spend their time has been consistent for decades: writing code accounts for roughly 30% of a typical developer's working hours. The rest is reading, understanding, planning, reviewing, communicating, debugging, and waiting. Optimizing the 30% while leaving the 70% untouched was always going to produce limited returns.

What's more, the 30% is the part that looks like productivity but often isn't the constraining factor. A developer who can write code twice as fast doesn't finish features twice as fast if the bottleneck is a design review that hasn't happened yet, a dependency that's blocked on another team, or a ticket description that requires three rounds of clarification before the work can begin.

The tools that generated the most enthusiasm — autocomplete, code generation, chat-based pair programming — all address the same sliver of the problem. They're genuinely useful for that sliver. But the sliver isn't the constraint.

---

## The Real Bottlenecks

Building software is a lifecycle, not a moment. It runs from idea to specification to implementation to review to deployment to monitoring and back around. Code generation occupies a narrow band in the middle of that arc. Everything upstream and downstream of it — the planning, the coordination, the validation, the iteration — runs largely the same as it did five years ago.

The downstream costs are now growing faster than before. When AI generates code at scale, the review surface expands. A developer who might have written 200 lines in a session can now produce 800. But the organization's capacity to review, test, and reason about those 800 lines hasn't quadrupled. It's grown modestly, if at all.

By some projections, 60% of new code will be AI-generated by the end of 2026. That's not a productivity number. That's a volume number. And volume without proportional review capacity doesn't produce faster software — it produces a larger backlog of unreviewed, partially understood, potentially insecure code that accumulates technical debt faster than teams can repay it.

The constraint isn't the code. It's the attention required to understand, verify, and integrate the code into a system that someone else will have to maintain.

---

## Context Switching and the Hidden Tax

There's another cost that rarely appears in productivity studies because it's hard to measure: context switching.

Modern software development is fragmented across tools. A feature might move through a product brief in Notion, a ticket in Linear, a branch in GitHub, a pull request with inline comments, a deployment preview, a Slack thread where someone flagged a concern, and a monitoring dashboard after launch. Each transition requires a developer to reload context — to remember where they were, what's been decided, what's still open.

AI tools, in their current form, mostly live at the code-generation layer. They don't hold the context of the broader lifecycle. They don't know that the function they just generated conflicts with a decision made two weeks ago in a design review that wasn't documented anywhere a language model can see. They don't know that the API endpoint they're scaffolding has a pending deprecation discussion in a Slack thread from last Thursday.

So the developer bridges these gaps manually, which means the cognitive load of the lifecycle hasn't decreased — it's just been redistributed. The typing got faster. The thinking didn't.

---

## What a Better Approach Looks Like

The tools that will actually move the needle aren't the ones that write the most code the fastest. They're the ones that understand the lifecycle and operate across its full span.

That means something more like a project layer than a coding layer. Something that holds context across planning, implementation, and review — that knows what was decided and why, that can flag when new code conflicts with existing constraints, that participates in the coordination overhead rather than just the typing overhead.

It means treating the developer's time as an attention budget, not a typing budget. The question isn't "how many lines can we generate per hour?" It's "how many decisions can a developer make well in a day, and how do we make sure each of those decisions is informed, traceable, and doesn't require them to reconstruct context from scratch?"

This is a harder problem than autocomplete. It requires understanding software as a social and organizational process, not just a technical one. The code is almost the easy part.

This is the bet we're making at Celune — that lifecycle-aware tooling will outperform generation-speed tooling once the industry fully reckons with where the real friction lives.

---

## The Lesson from the METR Study

The METR finding is uncomfortable for an industry that has spent three years confidently projecting productivity gains from AI tooling. But it's the kind of discomfort worth sitting with.

It doesn't say AI can't help. It says we've been measuring the wrong things, optimizing for the wrong layer, and drawing confident conclusions from proxies — code volume, generation speed, lines per session — that don't map cleanly to actual software delivery outcomes.

The developers in that study weren't made slower because the AI was bad at generating code. They were made slower because integrating AI-generated code into a real system, under real conditions, with real review requirements, takes more total effort than the generation step suggests. The work that happens after the cursor stops blinking hasn't gotten faster.

What would it look like to take that seriously? To build tools that treat review, planning, coordination, and context management as first-class problems rather than afterthoughts? To measure productivity in shipped, stable, maintainable features rather than lines of code per hour?

Those questions feel less exciting than a demo of a model that writes a REST API in thirty seconds. But they're the ones that matter. The bottleneck was never the typing. It was everything else. And that's where the opportunity actually lives.
